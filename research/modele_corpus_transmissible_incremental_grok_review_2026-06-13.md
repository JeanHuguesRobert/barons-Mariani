# Constructive Review: Incremental Transmissible Corpus Model

This file archives the core actionable review received from Grok for `research/modele_corpus_transmissible_incremental.md` v0.2. The full uploaded review remains the primary source in the conversation; this corpus file preserves the signal/noise conclusions selected for integration.

## Summary of the review

The review finds the document already usable as a speculative source-hypothesis. It confirms the central thesis: current LLMs lack a durable, versioned, stigmergic memory of exploration itself. The proposed Incremental Transmissible Corpus Model is read as a system where a sovereign versioned corpus is compiled into an operational layer: exploration graph, typed cognitive units, qualitative stigmergic traces, vector indexes, verifiers and scheduler heuristics. The LLM remains a non-sovereign probabilistic navigator.

The review considers the document medium-high in symmetry for a speculative source document. The core distinctions are recoverable from the text alone: source corpus sovereignty, tree-like exploration becoming graph, LLM as heuristic, qualitative stigmergic traces, scheduler as cognitive-exergy allocator, evidential levels, and explicit human validation.

## Stabilized concepts ready for reuse

- Source corpus sovereignty.
- LLM as non-sovereign heuristic.
- Qualitative stigmergic trace.
- Typed cognitive units.
- Exploration graph as first-class citizen.
- Scheduler / cognitive exergy allocation.
- Evidential levels and human validation anchor.
- Self-containment principle.
- Speculative embryonic status.

## Fragile or ambiguous concepts

- The term "Incremental Transmissible Corpus Model" may create a category error if the model does not yet implement inference, navigation, update and validation functions.
- Node granularity and branch equivalence remain the main feasibility blockers.
- "Cognitive exergy" must be defined or replaced by a more operational term.
- Trace validation workflow is not yet sufficiently specified.
- Derivative product synchronization is asserted but not operationalized.
- Scheduler operationalization remains a list of criteria rather than a minimal decision procedure.

## Strong objections to integrate

1. Documentary overhead may suffocate exploration if tracing becomes more costly than thinking.
2. Branch equivalence, fusion and contradiction detection are substantially harder than the document implies.
3. Governance and long-term maintenance are under-specified beyond single-author use.
4. The acceleration claim remains untested and requires baseline comparison.
5. Architectural detail can create premature closure pressure.

## High-signal integration recommendations

- Add a visible box of key open questions after the abstract.
- Add an explicit methodological alignment note with `second_method.md`.
- Define cognitive exergy in one sentence or replace it.
- Make the scheduler central earlier in the document.
- Convert paths to explore into a prioritized roadmap with success criteria.
- Move detailed YAML examples to an annex.
- Add a first branch-relation taxonomy: strong equivalence, partial convergence, analogy, neighborhood, apparent contradiction, real contradiction.
- Add a trace validation workflow: proposed -> evidence_checked -> human_or_rule_validated -> reusable, with rejected and superseded statuses.
- Keep the document in `barons-Mariani/research/` until a technical derivative is justified in `cogentia` or `inseme`.

## Recommended next action

Run a micro-prototype on a small curated set of documents. Deliverables:

2. one populated graph fragment;
3. simple scheduler stub;
4. short lessons-learned note on overhead versus reuse.

The reviewer recommends not moving the document to `cogentia` or `inseme` before such prototype feedback.
