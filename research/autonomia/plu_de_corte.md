# 0.3. Issues GitHub comme voie normale de contribution

La voie recommandée pour contribuer à l’enquête est l’ouverture d’une **Issue GitHub**.

Dans GitHub, une Issue est un fil structuré rattaché à un dépôt. Elle peut servir à signaler un problème, mais aussi à proposer une idée, documenter une source, ouvrir une piste, formuler une objection, suivre une tâche ou discuter une amélioration.

Dans cette enquête, une **Issue GitHub** doit être comprise comme une **unité contributive** : un conteneur public, daté, traçable, discutable et imputable.

Les Issues GitHub sont préférables aux commentaires dispersés sur les réseaux sociaux parce qu’elles permettent :

- de formuler une piste ;
- de documenter une source ;
- de dater une recherche ;
- de discuter une objection ;
- de conserver une trace publique ;
- de relier une contribution à une continuation ;
- de transformer une intuition en hypothèse vérifiable ;
- de préparer une éventuelle intégration dans le document source.

Principe :

> Les réseaux sociaux peuvent signaler une intuition ou un désaccord. Les Issues GitHub doivent devenir le lieu de transformation de cette intuition en élément d’enquête rationnellement discutable, daté et imputable.

Format recommandé d’une Issue :

```markdown
## Objet

## Source ou fait allégué

## Date de la source ou de la recherche

## Auteur ou contributeur identifiable

## Mandat ou qualité, si contribution au nom d’une personne morale

## Niveau de preuve
- [ ] fait établi
- [ ] fait rapporté
- [ ] trace publique
- [ ] absence documentaire
- [ ] indice
- [ ] hypothèse
- [ ] objection
- [ ] correction

## Objection possible

## Continuation liée
Ex. C01, C02, C03...

## Suite proposée
```

Les Issues ne sont pas des jugements.  
Elles sont des **conteneurs de mémoire en tension**.

# 0.4. GitHub comme infrastructure de traçabilité

Dans cette enquête, GitHub n’est pas seulement utilisé comme espace de publication ou de stockage documentaire.

Il est utilisé comme **infrastructure de traçabilité**.

Cette fonction est centrale dans le corpus général de l’Autonomie de Capacité : une action publique, une contribution, une correction, une objection ou une décision ne devient réellement contrôlable que si elle laisse une trace.

Formule directrice :

> Sans trace, aucune imputabilité n’est possible.  
> Sans imputabilité, la transparence reste déclarative.

Les **Issues GitHub**, commits, pull requests, commentaires, historiques de modification, auteurs identifiés et horodatages permettent de conserver la mémoire des contributions et des transformations du dossier.

Ils permettent notamment de savoir :

- qui a proposé une source ;
- qui a formulé une hypothèse ;
- qui a soulevé une objection ;
- qui a corrigé une erreur ;
- qui a validé une modification ;
- à quelle date une information a été ajoutée ;
- pourquoi une piste a été ouverte, maintenue, transformée ou abandonnée.

Cette traçabilité ne doit pas être confondue avec une surveillance généralisée. Elle vise l’imputabilité des actes publics ou contributifs, non l’exposition inutile des personnes.

Principe :

> Les contributions doivent être imputables à des personnes physiques identifiables chaque fois que cela est possible.

Les contributions issues de personnes morales — associations, institutions, collectivités, entreprises — peuvent être utiles, mais elles doivent être strictement encadrées : il faut autant que possible identifier la personne physique qui s’exprime ou agit au nom de cette personne morale, ainsi que son mandat, sa fonction ou son périmètre d’intervention.

Cette exigence protège le dossier contre trois risques :

1. l’anonymat irresponsable ;
2. la parole institutionnelle sans auteur identifiable ;
3. la dilution de responsabilité derrière une organisation.

Dans cette perspective, une **Issue GitHub** n’est pas seulement une question, un signalement ou un “problème”. C’est une **unité contributive traçable** : un conteneur public, daté, discutable et imputable, permettant de transformer une intuition, une source, une objection ou une correction en élément rationnellement examinable.

Chaîne doctrinale :

```text
trace
  -> imputabilité
  -> responsabilité
  -> correction possible
  -> confiance vérifiable
```

# 0.7. Rattachement au corpus : impunité par obscurité

Cette enquête s’inscrit dans le prolongement du travail sur l’**impunité par obscurité**.

Un outil fermé d’aide à la décision publique appliqué au bâti, au foncier et au logement étudiant peut créer une situation où les décisions finales sont formellement visibles, mais où leurs critères préparatoires restent invisibles.

Dans ce cas, la difficulté n’est pas seulement l’absence de transparence. C’est l’impossibilité pratique d’imputer les responsabilités :

- qui a choisi les données ?
- qui a choisi les critères ?
- qui a pondéré les variables ?
- qui a validé les scores ?
- qui a accès aux cartes ?
- qui peut corriger les erreurs ?
- qui répond si un outil influence une décision publique ?

L’enquête ne présume pas que Riacquistu produit cette situation.

Elle vérifie les conditions permettant de l’éviter.

Formule :

> Là où les critères préparatoires deviennent invisibles, l’imputabilité publique devient fragile.

# 0.8. Résumé d’orientation

Riacquistu Data-Driven est présenté publiquement comme un outil d’intelligence territoriale appliqué au bâti ancien cortenais, avec cartographie, monitoring, scoring, scénarios de réhabilitation et mise à disposition de la mairie et de l’Observatoire territorial du logement étudiant.

L’enjeu de ce dossier n’est pas de contester l’usage des données territoriales. L’enjeu est de vérifier les conditions de transparence, de traçabilité, d’auditabilité, de correction et d’imputabilité d’un outil susceptible d’éclairer des décisions publiques sur le bâti, le foncier et le logement étudiant.

Le risque examiné n’est pas une faute présumée. C’est une architecture possible d’**impunité par obscurité** : les décisions finales peuvent être publiques, tandis que les données, critères, pondérations, scores et cartes qui les préparent restent invisibles.

La question centrale est donc :

> Qui voit la carte ? Qui choisit les critères ? Qui peut corriger les erreurs ? Qui répond des effets produits ?

Ce document répond par une méthode : OSINT, niveaux de preuve, hypothèses et contre-hypothèses, continuations, GitHub comme infrastructure de traçabilité, Issues comme unités contributives imputables.

# 5. Qu’est-ce qu’une enquête OSINT ?

OSINT signifie **Open Source Intelligence**, ou renseignement d’origine sources ouvertes.

Dans ce dossier, l’OSINT désigne la recherche, la collecte, le recoupement et l’analyse d’informations publiquement accessibles :

- documents municipaux ;
- données INSEE ;
- documents DREAL, AUE, ADIL, CROUS, Université de Corse ;
- presse locale et nationale ;
- bases ouvertes comme DVF ;
- publications LinkedIn publiques ;
- profils professionnels publics ;
- documents d’urbanisme ;
- rapports institutionnels ;
- sites d’acteurs publics ou parapublics ;
- publications techniques ou académiques.

L’OSINT ne doit pas être confondu avec l’code source ouvert logiciel. Ici, “source ouverte” signifie source d’information accessible publiquement.

Une enquête OSINT sérieuse ne transforme pas une impression en conclusion. Elle transforme une impression en hypothèse, puis cherche :

- des sources ;
- des contradictions ;
- des objections ;
- des recoupements ;
- des absences documentaires datées ;
- des réfutations possibles.

Dans ce dossier, une absence publique d’information n’est jamais traitée seule comme une preuve de faute. Elle est traitée comme une **absence documentaire** à dater, qualifier, vérifier et, si nécessaire, transformer en demande institutionnelle.

## 6.2. Méthode Cogentia

Le dossier suit la méthode Cogentia :

```text

  -> pistes ouvertes
  -> continuations
  -> checkpoint
  -> document source
  -> produits déclinés
  -> retours
  -> réintégration au corpus
```

Une continuation n’est pas encore une conclusion.  
C’est une piste structurée qui garde en mémoire :

- une question ;
- un angle mort ;
- des sources à chercher ;
- des objections possibles ;
- un résultat attendu ;
- un critère de stabilisation ou d’abandon.

## 6.3. Méthode des terrains féconds

L’enquête refuse le terrain stérile :

> “Riacquistu est une manipulation” contre “Riacquistu est une innovation vertueuse”.

Elle propose un terrain plus fécond :

> Les acteurs affirment produire de l’observation, de l’objectivation et de l’aide à la décision ; l’enquête vérifie les conditions de validité, d’auditabilité et de correction de cette observation.

Cette méthode ne disqualifie pas les observateurs.  
Elle vérifie si leurs observations sont elles-mêmes observables, discutables, auditables et corrigibles.

Formule :

> Prendre les observateurs au sérieux, c’est observer leurs conditions d’observation.

## 6.4. Autonomie de Capacité

L’enquête évalue les outils territoriaux selon leur effet sur les capacités réelles :

- se loger ;
- connaître le parc bâti ;
- réhabiliter ;
- décider ;
- contrôler ;
- corriger ;
- empêcher la capture ;
- transmettre une méthode réutilisable.

La question centrale n’est donc pas seulement :

> L’outil est-il légal ou utile ?

mais aussi :

> Rend-il Corte, les étudiants, les habitants, les propriétaires modestes et les institutions plus capables, ou rend-il seulement certains acteurs mieux informés ?

## 6.5. Niveaux de preuve

| Niveau | Définition |
|---|---|
| Fait établi | Élément appuyé par une source officielle, institutionnelle ou documentaire identifiable. |
| Fait rapporté | Élément rapporté par la presse ou une publication publique, à recouper si nécessaire. |
| Trace publique | Post, repartage, commentaire, réaction ou profil public. |
| Absence documentaire | Élément attendu mais non trouvé après recherche datée. |
| Indice | Élément compatible avec une hypothèse, mais insuffisant seul. |
| Hypothèse | Proposition de travail à tester. |
| Objection | Élément pouvant affaiblir ou déplacer une hypothèse. |
| Réfutation | Élément invalidant l’hypothèse. |
| Interprétation | Lecture possible de plusieurs faits, séparée des faits eux-mêmes. |

## 6.6. Principe de prudence

Le dossier ne confond pas :

- absence documentaire et preuve d’absence ;
- interaction LinkedIn et implication opérationnelle ;
- intérêt situé et conflit d’intérêts ;
- conflit potentiel et conflit avéré ;
- hypothèse de capture et accusation personnelle ;
- transparence totale et transparence adaptée.

Formule de méthode :

> Pas de conclusion sans preuve ; pas de confiance sans méthode vérifiable.

# 7. Cadrage par les terrains féconds

## 7.1. Terrain stérile à éviter

Le terrain stérile serait :

> “Les institutions manipulent” contre “les institutions objectivent.”

Ce terrain force chacun à se défendre ou à se coaliser. Il produit vite un pugilat : accusation contre communication institutionnelle.

## 7.2. Terrain fécond proposé

Le terrain fécond proposé est :

> Si Riacquistu Data-Driven, l’OTLE et les autres dispositifs d’observation territoriale se présentent comme des outils d’objectivation, d’aide à la décision et de pilotage public, quelles conditions doivent-ils remplir pour que cette objectivation soit vérifiable, discutable et utile au territoire ?

L’enquête prend donc les acteurs au mot.

Elle ne rejette pas l’observation territoriale.  
Elle l’applique à elle-même.

Un observatoire qui observe le territoire doit pouvoir être observé dans sa méthode.

Une carte qui éclaire la décision publique doit pouvoir être éclairée dans ses sources, ses critères, ses limites et ses effets.

Un notation qui classe des situations doit pouvoir être discuté comme tout instrument de classement.

## 7.3. Tableau de reconfiguration

| Terrain stérile | Terrain fécond |
|---|---|
| Accuser ou défendre Riacquistu | Examiner les conditions de validité d’un outil d’observation territoriale |
| Pour ou contre la data | Quelles données, selon quelles règles, avec quels droits d’accès et de correction ? |
| Innovation contre soupçon | Innovation vérifiable, auditable, corrigible |
| Communication institutionnelle contre dénonciation | Enquête méthodique sur les sources, critères, effets et absences |
| Faire confiance ou ne pas faire confiance | Rendre la confiance vérifiable |

# 20. Hypothèses de travail

## H0 — Hypothèse neutre minimale

Riacquistu Data-Driven est un simple POC pédagogique, sans usage décisionnel réel.

À vérifier :

- absence d’intégration au processus municipal ;
- absence de convention opérationnelle ;
- absence de maintenance ;
- absence de décisions préparées ou orientées par l’outil.

Éléments qui affaiblissent déjà H0 :

- le projet est présenté comme mis à disposition de la mairie et de son Observatoire territorial du logement étudiant ;
- EONA-X revendique l’usage de “notre technologie” ;
- les réactions et repartages dépassent le cercle étudiant.

## H1 — Hypothèse institutionnelle faible

Riacquistu est un POC utile, fermé par défaut, sans intention problématique, mais dont la publication, la documentation et la gouvernance n’ont pas encore été organisées.

À vérifier :

- publication future des Cahiers du Riacquistu ;
- documentation prévue ;
- version publique ou anonymisée envisagée ;
- clarification RGPD ;
- clarification de la propriété intellectuelle ;
- accès possible aux chercheurs, citoyens ou associations.

## H2 — Hypothèse de boîte noire municipale

Riacquistu devient un outil interne d’aide à la décision sur le bâti, le logement et le foncier, sans accès public aux données, critères, scores, cartes ou scénarios.

Indices compatibles à ce stade :

- absence d’URL publique trouvée ;
- absence de dépôt public trouvé ;
- absence des Cahiers du Riacquistu trouvés ;
- vocabulaire de mise à disposition de la mairie, non du public ;
- insertion dans des enjeux majeurs du PLU : bâti ancien, logement étudiant, ORT, OPAH, centre ancien, ZAN.

## H3 — Hypothèse d’asymétrie informationnelle foncière

L’outil produit une connaissance stratégique du bâti et du foncier — vacance, dégradation, potentiel, indivisions, opportunités — accessible à certains acteurs mais non aux citoyens, propriétaires, étudiants ou associations.

À vérifier :

- nature exacte du lac de données ;
- présence de données sur vacance, indivisions, péril, potentiel de réhabilitation ;
- acteurs ayant accès ;
- droits de consultation, correction ou contestation par les propriétaires ;
- liens avec ORT, OPAH, projets immobiliers et politique du logement étudiant.

## H4 — Hypothèse forte : outil favorable au statu quo ou à la capture de valeur

L’outil pourrait, volontairement ou de fait, favoriser le statu quo et la capture spéculative de la valeur dans un contexte de pénurie structurelle de logements étudiants à Corte.

État actuel :

> Hypothèse non démontrée.

Chemin probatoire réaliste :

1. opacité de l’outil ;
2. asymétrie d’accès ;
3. absence de critères publics ;
4. absence de contestabilité ;
5. concentration des bénéfices informationnels ;
6. décisions ou projets immobiliers postérieurs alignés avec les zones identifiées ;
7. exclusion des étudiants, propriétaires modestes, associations ou habitants de la chaîne de décision.

Formulation prudente :

> L’architecture peut produire des effets de capture, même si l’intention de capture n’est pas établie.

## H5 — Hypothèse de dépolitisation des étudiants comme acteurs de gouvernance

Les étudiants peuvent être présents dans le discours comme bénéficiaires — à loger, aider, nourrir, accompagner — sans être associés comme acteurs de gouvernance aux outils territoriaux qui prétendent répondre à leurs besoins.

À vérifier :

- élus étudiants CROUS ;
- syndicats étudiants ;
- PV de CA CROUS ;
- conseils universitaires ;
- documents OTLE ;
- consultations PLU ;
- accès des représentants étudiants aux données ou aux résultats ;
- mentions publiques d’une consultation sur Riacquistu.

## H6 — Hypothèse capacitaire constructive

Riacquistu pourrait devenir un commun territorial capacitaire si ses données, critères, cartes, cahiers, méthodes et résultats étaient publiés sous une forme adaptée, respectueuse des données sensibles, mais suffisamment ouverte pour permettre l’audit, la correction, la réutilisation et la transmission.

Conditions minimales possibles :

1. publication des Cahiers du Riacquistu ;
2. publication de la méthodologie ;
3. publication des critères de notation ;
4. publication des pondérations ou justification de leur confidentialité ;
5. dépôt public du code ou des briques réutilisables ;
6. version publique anonymisée de la cartographie ;
7. registre des données utilisées ;
8. droit de correction pour les propriétaires concernés ;
9. accès des représentants étudiants ;
10. documentation permettant la réplication dans d’autres communes corses.

# 21. Contre-hypothèses loyales

Pour éviter un dossier à charge, les contre-hypothèses suivantes doivent être explicitement intégrées.

## CH1 — POC pédagogique sans usage décisionnel

Riacquistu n’a peut-être jamais été utilisé pour décider. Il serait alors un exercice étudiant mal publicisé, non une boîte noire municipale.

## CH2 — Confidentialité légitime

Certaines données ne peuvent peut-être pas être publiées : données propriétaires, données sensibles, risques de spéculation, sécurité, RGPD.

## CH3 — Publication différée

Les Cahiers, cartes ou méthodes peuvent être en préparation, non encore publiés.

## CH4 — Outil interne mais non opposable

L’outil peut aider à réfléchir sans produire d’effet sur les droits, biens ou priorités.

## CH5 — Absence de communication non intentionnelle

L’absence de communication peut venir d’un défaut de culture de publication, d’un retard, ou d’une négligence administrative.

## CH6 — Consultation étudiante non visible

Les étudiants ou leurs représentants ont peut-être été consultés via questionnaire, CROUS, Université, OTLE ou réunions non visibles.

# 22. Problème démocratique central

Le point critique n’est pas l’existence d’un outil de données. Un tel outil peut être utile, et même nécessaire.

Le problème apparaît si l’outil :

- prépare des décisions publiques ;
- classe, score ou priorise des biens ;
- produit une cartographie stratégique du bâti ;
- influence les politiques de logement, d’urbanisme ou de rénovation ;
- reste inaccessible au public ;
- ne publie ni son code, ni ses critères, ni ses pondérations, ni ses cahiers, ni ses cartes.

Dans ce cas, il ne produit pas nécessairement une transparence démocratique. Il peut produire une **visibilité asymétrique**.

Formulation de travail :

> Un outil fermé d’aide à la décision publique, portant sur le bâti ancien, le foncier et le logement étudiant dans une ville sous tension, crée objectivement une asymétrie informationnelle au bénéfice des acteurs qui y ont accès.

# 23. Grille capacitaire de l’enquête

Cette enquête n’a pas seulement pour objet de vérifier si Riacquistu Data-Driven est accessible, code source ouvert ou documenté.

Elle vise à mesurer si cet outil augmente ou réduit l’Autonomie de Capacité du territoire.

Une capacité publique est ici évaluée selon sept critères :

| Critère | Question appliquée à Riacquistu |
|---|---|
| Capacité réelle | Que permet concrètement l’outil : loger plus d’étudiants, réhabiliter plus vite, réduire la vacance, prioriser mieux ? |
| Traçabilité | Peut-on suivre les données, les critères, les versions, les accès, les décisions produites ? |
| Imputabilité | Qui répond des erreurs, biais, oublis, priorisations ou effets de capture ? |
| Contrôlabilité | Les citoyens, élus, étudiants, propriétaires ou chercheurs peuvent-ils auditer, corriger ou contester ? |
| Anti-capture | Quels garde-fous empêchent la captation par intérêts fonciers, technocratiques, clientélistes, spéculatifs ou institutionnels ? |
| Transmission | L’outil produit-il un commun réutilisable, documenté, transmissible à d’autres communes ? |
| Situation territoriale | L’outil tient-il compte de Corte comme ville universitaire, ville de montagne, ville patrimoniale, ville touristique et ville sous tension foncière ? |

Question centrale :

> Riacquistu rend-il Corte plus capable, ou rend-il seulement certains acteurs mieux informés ?

# 24. Capacités en jeu

## 24.1. Capacité de se loger

Capacité pour les étudiants, familles, jeunes actifs, saisonniers sobres et habitants permanents à accéder à un logement digne, stable et compatible avec leurs revenus.

## 24.2. Capacité de connaître le parc bâti

Capacité collective à connaître le bâti vacant, dégradé, sous-utilisé, réhabilitable ou menacé.

## 24.3. Capacité de réhabiliter

Capacité à transformer du bâti ancien en logements réellement habitables, sans détruire le patrimoine ni produire seulement de la rente.

## 24.4. Capacité de décider publiquement

Capacité à décider à partir de données fiables, accessibles, discutables et traçables.

## 24.5. Capacité de contrôler

Capacité pour les citoyens, élus d’opposition, étudiants, propriétaires, associations et chercheurs à vérifier les outils qui préparent les décisions publiques.

## 24.6. Capacité d’empêcher la capture

Capacité à empêcher que les données, cartes, scores ou scénarios deviennent un avantage réservé aux acteurs déjà puissants.

## 24.7. Capacité de transmettre

Capacité à produire un commun territorial : données documentées, code ouvert, méthode réplicable, formation d’étudiants, savoir-faire local, mémoire des décisions.

# 28. Conflits d’intérêts et intérêts situés

Le dossier ne présume pas l’existence de conflits d’intérêts fautifs.

Il constate seulement que tout observateur est situé.

Un acteur peut être à la fois :

- producteur de données ;
- bénéficiaire d’une politique publique ;
- financeur ;
- opérateur technique ;
- décideur ;
- propriétaire ;
- aménageur ;
- prestataire ;
- élu ;
- expert ;
- communicant ;
- bénéficiaire indirect de la valeur produite par l’information.

Distinctions proposées :

| Catégorie | Définition |
|---|---|
| Intérêt légitime | Intérêt normal lié à une mission, compétence ou responsabilité. |
| Intérêt situé | Position qui influence naturellement le regard porté sur le problème. |
| Conflit d’intérêts potentiel | Situation où deux intérêts peuvent entrer en tension. |
| Conflit d’intérêts avéré | Situation documentée où un intérêt compromet ou semble compromettre l’impartialité. |
| Capture | Situation où une capacité collective est confisquée au profit d’acteurs particuliers. |

Règle de prudence :

> Identifier un intérêt situé n’est pas accuser. C’est rendre la méthode plus robuste.

# 36. Comment contribuer à l’enquête

## 36.1. Principe : contribuer par Issues GitHub

La voie normale de contribution à cette enquête est l’ouverture d’une **Issue GitHub**.

Une Issue GitHub permet de transformer une remarque, une intuition, une objection, une source ou une correction en objet public, daté, discutable, traçable et imputable.

Ce dossier est une enquête OSINT ouverte sur le PLU de Corte, Riacquistu Data-Driven, le logement étudiant, le bâti ancien, les données territoriales et les conditions de transparence démocratique associées.

Il ne s’agit pas d’un espace de dénonciation vague, de procès d’intention ou de polémique personnelle.

L’objectif est de transformer des impressions, alertes, soupçons, intuitions, affirmations ou désaccords en éléments rationnellement discutables :

- faits documentés ;
- sources vérifiables ;
- hypothèses explicites ;
- objections formulées ;
- réfutations possibles ;
- niveaux de preuve ;
- pistes d’enquête ;
- demandes de clarification ;
- corrections du dossier.

Principe général :

> Une opinion peut ouvrir une piste. Elle ne devient un élément du dossier qu’après avoir été formulée comme hypothèse, sourcée, discutée, réfutable, imputable et située dans un niveau de preuve explicite.

## 36.2. GitHub, trace et imputabilité

L’usage de GitHub est directement lié à la doctrine de traçabilité du corpus.

Dans ce dossier :

- une Issue conserve la trace d’une piste ;
- un commentaire conserve la trace d’une objection ;
- une pull request conserve la trace d’une proposition de modification ;
- un commit conserve la trace d’une intégration ;
- l’historique Git conserve la trace des transformations successives ;
- les labels permettent de qualifier les contributions ;
- les dates et auteurs rendent les actes imputables.

Formule :

> Sans trace, aucune imputabilité n’est possible.  
> Sans imputabilité, la transparence reste déclarative.

L’enquête doit donc favoriser les contributions imputables à des personnes physiques identifiables.

Lorsqu’une contribution est faite au nom d’une personne morale — association, collectivité, institution, entreprise, collectif — elle doit autant que possible préciser :

- le nom de la personne physique contributrice ;
- sa fonction ;
- son mandat ;
- le périmètre de ce mandat ;
- l’organisation au nom de laquelle elle intervient ;
- ce qui relève d’une position personnelle et ce qui relève d’une position institutionnelle.

Cette règle n’a pas pour but d’exposer inutilement les personnes. Elle vise à éviter l’irresponsabilité, l’anonymat stratégique et la dilution de responsabilité derrière des structures abstraites.

## 36.3. Outils GitHub utilisables

Les contributions peuvent se faire par les outils GitHub habituels :

1. **Issues GitHub**  
   Voie recommandée pour ouvrir une piste, signaler une source, poser une question, formuler une objection, proposer une hypothèse ou documenter une absence.

2. **Commentaires d’Issues**  
   Voie recommandée pour discuter une piste sans modifier immédiatement le document source.

3. **Pull requests**  
   Pour proposer une modification directe du dossier : ajout de source, correction factuelle, reformulation, tableau, note méthodologique.

4. **Commentaires de pull requests**  
   Pour relire une modification proposée avant intégration.

5. **Commits**  
   Pour stabiliser des ajouts validés, avec message de commit explicite.

6. **Fichiers annexes**  
   Pour regrouper, si nécessaire, des tableaux, chronologies, listes de sources, fiches d’acteurs ou notes thématiques.

## 36.4. Typologie recommandée des Issues

| Type d’Issue | Usage |
|---|---|
| `source` | Signaler une source utile. |
| `absence-documentaire` | Documenter une information attendue mais non trouvée. |
| `hypothèse` | Formuler une hypothèse à tester. |
| `objection` | Contester ou nuancer une hypothèse. |
| `correction` | Corriger une erreur factuelle ou méthodologique. |
| `continuation` | Ouvrir une piste de recherche autonome. |
| `question-institutionnelle` | Préparer une question neutre à un acteur public ou institutionnel. |
| `donnée` | Proposer une analyse de données : INSEE, DVF, tourisme, PLU. |
| `juridique` | Examiner CADA, RGPD, documents administratifs, propriété intellectuelle. |
| `produit-décliné` | Préparer une publication, un article, une fiche ou un courrier dérivé du document source. |

## 36.5. Règle Issue → document

Une Issue ne modifie pas automatiquement le document source.

Chemin recommandé :

```text
Issue GitHub
  -> discussion
  -> sources
  -> objections
  -> niveau de preuve
  -> proposition de modification
  -> point de contrôle
  -> intégration éventuelle
  -> commit traçable
```

Cette règle évite deux dérives :

- transformer trop vite une intuition en texte stabilisé ;
- perdre des pistes utiles dans des conversations dispersées.

## 36.6. Règle de contribution

Toute contribution doit préciser autant que possible :

| Élément | Question à traiter |
|---|---|
| Fait allégué | Que dit-on exactement ? |
| Source | D’où vient l’information ? |
| Date | Quand la source a-t-elle été publiée ou consultée ? |
| Contributeur | Quelle personne physique porte la contribution ? |
| Mandat éventuel | Parle-t-elle en son nom ou au nom d’une personne morale ? |
| Acteurs concernés | Qui est concerné, sans extrapolation abusive ? |
| Niveau de preuve | Fait établi, indice, hypothèse, opinion, témoignage, absence documentaire ? |
| Objection possible | Qu’est-ce qui pourrait invalider ou nuancer l’affirmation ? |
| Suite proposée | Chercher, vérifier, corriger, demander, classer, abandonner ? |

## 36.8. Ce qui est accepté

Sont bienvenues :

- sources officielles ;
- articles de presse identifiables ;
- documents municipaux ;
- documents administratifs ;
- publications institutionnelles ;
- captures datées de pages publiques ;
- liens vers profils publics lorsque leur fonction est pertinente ;
- tableaux de données ;
- demandes CADA ou réponses administratives ;
- corrections factuelles ;
- objections argumentées ;
- hypothèses alternatives ;
- signalement d’une erreur ou d’une extrapolation.

## 36.9. Ce qui est refusé

Sont exclus du dossier :

- insultes ;
- insinuations personnelles ;
- accusations non sourcées ;
- attaques contre les étudiants ;
- données privées ;
- coordonnées personnelles ;
- rumeurs invérifiables ;
- généralités sans objet précis ;
- affirmations impossibles à discuter ;
- amalgames entre réaction LinkedIn et implication opérationnelle ;
- procès d’intention non appuyés par des faits.

Formule de travail :

> Pas de pugilat.  
> Pas d’omerta.  
> Pas d’accusation sans preuve.  
> Pas de confiance sans accès, sans méthode publiée et sans traçabilité.  
> Toute affirmation doit pouvoir devenir une hypothèse discutable, une objection vérifiable ou un fait sourcé, avec une trace imputable.

# 38.6. Auto-évaluation méthodologique

## Points solides

- Le dossier distingue faits, hypothèses, absences documentaires et interprétations.
- Les hypothèses fortes sont explicitement présentées comme non démontrées.
- Les contre-hypothèses loyales sont formulées.
- Les contributions futures sont orientées vers des Issues GitHub traçables.
- La méthode vise l’imputabilité plutôt que la dénonciation.
- Le document est autoporteur pour un lecteur qui ne connaît pas le corpus.

## Limites reconnues

- Le document est long et n’est pas destiné à une lecture rapide.
- Certaines données restent à vérifier ou à dater plus finement.
- Les sources LinkedIn doivent être traitées comme traces publiques, non comme preuves d’implication.
- Les documents PLU complémentaires doivent encore être recherchés systématiquement.
- Les données DVF de Corte n’ont pas encore été exploitées.
- Les questions RGPD, CADA, propriété intellectuelle et communicabilité doivent faire l’objet de notes dédiées.

## Justification du maintien en document unique

Le maintien en document unique est volontaire à ce stade.

Il permet de conserver la continuité entre :

- les faits ;
- les méthodes ;
- les hypothèses ;
- les acteurs ;
- les continuations ;
- les règles de contribution ;
- la doctrine de traçabilité.

Le découpage en sous-répertoire, annexes ou documents spécialisés devra intervenir lorsque les pistes produiront assez de matière autonome pour justifier cette séparation.
